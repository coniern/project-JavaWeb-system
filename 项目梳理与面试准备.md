# project-JavaWeb-system 项目梳理与面试准备

## 1. 项目一句话介绍
这是一个面向项目协作与发布管理的 Java Web 项目，当前已收敛为 `Spring Boot + Spring Security + JWT + MyBatis-Plus + H2/MySQL` 的前后端一体化演示版本，覆盖登录注册、项目管理、权限控制和静态管理端页面展示。

1. 传统 Java Web 版本
   使用 `Servlet + JSP + JDBC + MySQL`，实现了用户登录、注册、列表、删除、更新。
2. Mock 演示版本
   使用 `com.app.ServerApp` 和部分 Mock Controller，主要用于前端页面联调。
3. 当前已接通的数据库版主链路
   使用 `Spring Boot + Spring Security + JWT + MyBatis-Plus`，已真实跑通：
   - 登录
   - 注册
   - 获取当前用户
   - JWT 鉴权

## 6. 专业问题与回答要点
### 6.1 为什么不用原始 JDBC 继续做？
答题要点：
1. 原始 JDBC 可控性高，但样板代码多。
2. 连接、SQL 拼接、对象映射都要手写，维护成本高。
3. 旧版代码里直接拼 SQL，有 SQL 注入风险。
4. MyBatis-Plus 能显著降低 CRUD 成本，并保留 SQL 可控性。

### 6.2 旧版 JDBC 有什么问题？
答题要点：
1. SQL 直接字符串拼接，存在注入风险。
2. 连接管理粗糙，没有连接池。
3. 配置写死在代码里，环境切换困难。
4. 业务、SQL、连接逻辑耦合严重。

### 6.3 为什么选 JWT？
答题要点：
1. 无状态，适合前后端分离或接口化系统。
2. 认证信息放在 token 中，服务端不需要保存 Session。
3. 更适合后续扩展成多端访问。
4. 缺点是主动失效、续签、黑名单机制要额外设计。

### 6.4 Spring Security 在这个项目中做了什么？
答题要点：
1. 统一接口访问控制。
2. 登录时走 `AuthenticationManager` 做认证。
3. 请求进入后由 JWT 过滤器解析 token。
4. 将用户信息放入 `SecurityContext`，供控制器和权限注解使用。

### 6.5 为什么会出现角色前缀问题？
答题要点：
1. `hasRole('ADMIN')` 本质会匹配 `ROLE_ADMIN`。
2. 如果数据库里只存 `ADMIN`，就需要在加载 `GrantedAuthority` 时补前缀。
3. 这是 Spring Security 的约定，不统一会导致 403。

### 6.6 为什么要保留旧代码而不是直接删掉？
答题要点：
1. 旧代码本身也是业务资产。
2. 便于对照迁移和回溯需求。
3. 在重构过程中先隔离默认运行链，比一次性大删更稳妥。

### 6.7 为什么默认运行用 H2，而不是直接 MySQL？
答题要点：
1. 先保证项目开箱即跑，提高演示成功率。
2. H2 适合本地验证主链路。
3. 同时保留 MySQL profile，后续部署可以直接切换。
4. 这是工程上的“可演示性优先 + 生产环境可切换”思路。

### 6.8 这次你解决的最关键问题是什么？
答题要点：
1. 不是单一 bug，而是工程收敛问题。
2. 包括启动类冲突、接口路径漂移、权限配置错位、数据源初始化、静态页托管。
3. 最终目标是把“代码很多”变成“系统真的能跑”。
